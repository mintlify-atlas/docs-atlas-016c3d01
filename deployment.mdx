Control de Vacaciones ships with a production-ready `docker-compose.yml` that starts four services: a web server running Laravel Octane on FrankenPHP, a dedicated queue worker for email notifications, a MySQL 8.0 database, and a Redis 7 cache. This guide walks you through configuring your environment variables and bringing the stack up from a fresh clone.

## Architecture overview

The Docker stack is defined in `docker-compose.yml` and consists of four services:

| Service | Image | Role |
|---------|-------|------|
| `web` | Built from `Dockerfile` | Serves the Filament panel via FrankenPHP + Octane on port 8080 |
| `queue` | Built from `Dockerfile` | Runs `php artisan queue:work` to process email notification jobs |
| `redis` | `redis:7-alpine` | Cache store and queue backend |
| `db` | `mysql:8.0` | Primary relational database |

The `web` and `queue` services use the same two-stage Docker image. The builder stage compiles PHP extensions and Vite assets; the runtime stage uses `dunglas/frankenphp:1.4-php8.4-alpine` with Laravel Octane. Both services read configuration from a shared `.env` file and mount a shared `storage_data` volume so uploaded files are consistent between the web and queue containers.

## Prerequisites

- Docker Engine 24+ and Docker Compose v2
- Git

## Step 1: Clone the repository

```bash
git clone https://github.com/it-hvd/control-vacaciones.git
cd control-vacaciones
```

## Step 2: Configure environment variables

Copy the example file and update the values for your environment:

```bash
cp .env.example .env
```

The variables most relevant to a Docker deployment are listed below. Update each one before starting the stack.

```bash
# Application
APP_NAME="Control de Vacaciones"
APP_ENV=production
APP_KEY=                          # Generated by php artisan key:generate
APP_DEBUG=false
APP_URL=http://your-domain.com

# Database — must match the db service credentials in docker-compose.yml
DB_CONNECTION=mysql
DB_HOST=db
DB_PORT=3306
DB_DATABASE=app
DB_USERNAME=app
DB_PASSWORD=secret

# Redis — host name matches the redis service name in docker-compose.yml
REDIS_CLIENT=phpredis
REDIS_HOST=redis
REDIS_PASSWORD=null
REDIS_PORT=6379

# Queue and cache — use Redis in production
QUEUE_CONNECTION=redis
CACHE_STORE=redis
SESSION_DRIVER=redis

# Mail — configure your SMTP provider for email notifications
MAIL_MAILER=smtp
MAIL_HOST=smtp.example.com
MAIL_PORT=587
MAIL_USERNAME=your@email.com
MAIL_PASSWORD=your-password
MAIL_FROM_ADDRESS="noreply@your-domain.com"
MAIL_FROM_NAME="Control de Vacaciones"
```

<Warning>
  The default `docker-compose.yml` sets `MYSQL_PASSWORD=secret` and `MYSQL_ROOT_PASSWORD=rootsecret`. Change both passwords before deploying to any environment accessible from the internet.
</Warning>

<Note>
  The default `.env.example` uses `DB_CONNECTION=sqlite`. You must switch to `mysql` and set the `DB_HOST=db` to point to the `db` service when running under Docker Compose.
</Note>

## Step 3: Review the docker-compose.yml

The full compose file for reference:

```yaml
services:

  web:
    build:
      context: .
      dockerfile: Dockerfile
    restart: always
    env_file:
      - .env
    depends_on:
      - db
      - redis
    volumes:
      - storage_data:/app/storage
    ports:
      - "8080:8080"  # Puerto administrativo
    networks:
      - default

  queue:
    build:
      context: .
      dockerfile: Dockerfile
    restart: always
    command: ["php", "artisan", "queue:work", "--tries=3", "--timeout=120", "--sleep=1", "--max-time=3600"]
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    volumes:
      - storage_data:/app/storage
    networks:
      - default

  redis:
    image: redis:7-alpine
    restart: always
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 10
    networks:
      - default

  db:
    image: mysql:8.0
    restart: always
    environment:
      MYSQL_DATABASE: app
      MYSQL_USER: app
      MYSQL_PASSWORD: secret
      MYSQL_ROOT_PASSWORD: rootsecret
    command: --default-authentication-plugin=mysql_native_password
    volumes:
      - db_data:/var/lib/mysql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 5s
      timeout: 5s
      retries: 10
    networks:
      - default

volumes:
  db_data:
  storage_data:
```

<Info>
  The `queue` service waits for both `db` and `redis` to pass their health checks before starting. The `web` service has a `depends_on` without a health condition — MySQL and Redis may still be initializing when the first HTTP request arrives, so run migrations after the stack is up (Step 5 below).
</Info>

## Step 4: Build and start the stack

```bash
docker compose up -d --build
```

Docker builds the image (this takes a few minutes on first run), then starts all four services. Check that all containers are running:

```bash
docker compose ps
```

You should see `web`, `queue`, `redis`, and `db` with a `running` or `healthy` status.

## Step 5: Run initial setup

Once all services are up, run the setup commands inside the `web` container. These match the `composer setup` script defined in `composer.json`:

<CodeGroup>

```bash One-liner (matches composer setup script)
docker compose exec web sh -c "
  php artisan key:generate &&
  php artisan migrate --force &&
  php artisan db:seed --force
"
```

```bash Step by step
# Generate the application key (writes to APP_KEY in .env)
docker compose exec web php artisan key:generate

# Run all database migrations
docker compose exec web php artisan migrate --force

# Seed roles, permissions, and the initial super_admin user
docker compose exec web php artisan db:seed --force
```

</CodeGroup>

The seeder (`database/seeders/DatabaseSeeder.php`) does three things:

1. Generates Filament Shield permissions for all resources, pages, and widgets in the `control_de_vacaciones` panel.
2. Creates a `super_admin` role with all permissions.
3. Creates an initial user at `admin@admin.com` with password `admin` and assigns the `super_admin` role.

<Warning>
  Change the default `admin@admin.com` / `admin` credentials immediately after seeding. You can do this from the **Users** section in the admin panel or via Tinker.
</Warning>

## Step 6: Access the panel

Open your browser and navigate to:

```
http://your-domain.com:8080/admin
```

Log in with `admin@admin.com` / `admin` to verify the installation.

## Queue worker

Email notifications (approved, rejected, pending) are dispatched as queued jobs via the `App\Mail` namespace. The `queue` service handles these automatically using:

```
php artisan queue:work --tries=3 --timeout=120 --sleep=1 --max-time=3600
```

The worker restarts itself every 3600 seconds (`--max-time=3600`) and Docker's `restart: always` policy brings it back up automatically. If a job fails after 3 attempts it is moved to the `failed_jobs` table.

<Tip>
  Monitor the queue with `docker compose logs -f queue` to see job processing in real time.
</Tip>

## Storage persistence

Both `web` and `queue` mount the same `storage_data` named volume at `/app/storage`. This ensures that files written by the web process (logs, framework cache, uploaded files) are readable by the queue worker and survive container restarts.

The `db_data` volume persists MySQL data across restarts and image rebuilds.
